# StoryFlow 提示词设计文档

## 概述

本文档说明 StoryFlow 项目中各个节点提示词模板的设计思路、设计原则和使用方法。

## 设计原则

所有提示词模板遵循以下核心原则：

1. **清晰性**: 明确任务目标、输入输出格式
2. **具体性**: 提供详细的操作指导和检查清单
3. **可操作性**: 给出具体的执行步骤
4. **参数化**: 支持灵活的参数替换
5. **结构化输出**: 便于节点间的数据传递
6. **质量保障**: 内置质量检查机制

---

## 1. 世界观设定节点 (worldview.txt)

### 设计目标
为故事构建完整、自洽、富有吸引力的世界观基础，为后续所有节点提供统一的设定框架。

### 核心设计思路

**为什么这样设计？**

世界观是小说的基石，它决定了：
- 故事发生的环境和规则
- 角色行动的边界和可能性
- 情节发展的合理性依据
- 读者的沉浸感和可信度

**设计决策：**

1. **五个维度覆盖**
   - 宇宙观：世界的本质和起源
   - 力量体系：能力的规则和限制
   - 地理环境：空间的布局和特色
   - 社会结构：组织和关系网络
   - 历史背景：时间的脉络和冲突

   这五个维度形成完整的世界框架，避免遗漏关键要素。

2. **参数化设计**
   - `{genre}`：决定世界观的基本类型
   - `{theme}`：影响设定的主题倾向
   - `{tone}`：控制整体氛围风格
   - `{world_scope}`：确定世界规模大小
   - `{magic_system}`：力量体系类型

   参数化使提示词能够适应不同类型的故事需求。

3. **结构化输出格式**
   使用 YAML 格式，便于：
   - 后续节点提取特定信息
   - 保持信息的结构化
   - 支持程序化处理

4. **质量检查清单**
   包含6项自检指标，确保：
   - 设定的自洽性
   - 逻辑的严密性
   - 故事的可扩展性

### 使用场景
- 故事开始阶段，建立世界观基础
- 类型切换时，创建新世界观
- 世界观需要重构或扩展时

### 输出数据结构
```yaml
worldview:
  universe: {宇宙观设定}
  power_system: {力量体系}
  geography: {地理环境}
  society: {社会结构}
  history: {历史背景}
  key_settings:
    uniqueness: {独特性}
    conflict: {核心矛盾}
    entry_point: {故事切入点}
```

### 与其他节点的关系
- **依赖于**: 无（首个节点）
- **输出给**: 角色档案、大纲结构、场景描写、章节写作、编辑修订

## 2. 角色档案节点 (character.txt)

### 设计目标
塑造立体、鲜活、有深度的角色，为故事提供驱动力和情感共鸣点。

### 核心设计思路

**为什么这样设计？**

角色是故事的灵魂。好的角色能够：
- 推动情节发展
- 引发读者情感共鸣
- 承载故事主题
- 创造冲突和张力

**设计决策：**

1. **六层角色模型**
   - 基本信息层：身份和外在特征
   - 性格与心理：内在驱动力和障碍
   - 能力与技能：影响行动的能力边界
   - 人际关系：社会关系和情感网络
   - 背景故事：塑造角色的经历
   - 独特特征：区分度与记忆点

   这六层由表及里，完整呈现角色全貌。

2. **角色弧光规划**
   - 初始状态 → 成长路径 → 终极形态
   - 明确角色的成长方向
   - 为情节发展提供动力

3. **参数化设计**
   - `{character_name}`：角色标识
   - `{role}`：故事中的定位
   - `{alignment}`：道德倾向
   - `{background}`：出身背景
   - `{worldview_context}`：继承世界观

4. **质量检查清单**
   包含7项自检指标，确保：
   - 角色的立体性
   - 行为的合理性
   - 与世界观的契合度
   - 成长的可能性

### 使用场景
- 故事规划阶段，创建主要角色
- 新角色登场时，建立档案
- 角色发展需要重构时

### 输出数据结构
```yaml
character:
  basic_info: {基本信息}
  personality: {性格特征}
  abilities: {能力设定}
  relationships: {人际关系}
  backstory: {背景故事}
  character_arc: {角色弧光}
  unique_features: {独特设定}
```

### 与其他节点的关系
- **依赖于**: 世界观设定
- **输出给**: 大纲结构、场景描写、章节写作、编辑修订

## 3. 大纲结构节点 (outline.txt)

### 设计目标
规划故事的宏观结构，确保情节推进合理、节奏控制得当、主题贯穿始终。

### 核心设计思路

**为什么这样设计？**

大纲是故事的骨架，它能：
- 提供清晰的创作路线图
- 控制故事的整体节奏
- 确保情节的连贯性
- 避免创作过程中的迷失

**设计决策：**

1. **关键转折点设计**
   - 激励事件：故事启动点
   - 第一幕高潮：问题确立
   - 中点转折：情势改变
   - 第二幕高潮：困境加深
   - 第三幕高潮：最终对决

   这些转折点确保故事有明确的推进节奏。

2. **章节分层规划**
   - 开篇阶段：引入和铺垫
   - 发展阶段：矛盾升级
   - 高潮阶段：冲突爆发
   - 结局阶段：收束和升华

   分阶段规划便于控制节奏和分配笔墨。

3. **支线与伏笔系统**
   - 支线故事：丰富世界观和角色
   - 伏笔线索：增强故事的层次感
   - 冲突升级：保持读者的紧张感

4. **参数化设计**
   - `{worldview_context}`：世界观基础
   - `{character_context}`：角色驱动
   - `{total_chapters}`：篇幅控制
   - `{pacing}`：节奏风格
   - `{structure}`：结构类型
   - `{theme}`：核心主题

5. **质量检查清单**
   包含8项自检指标，确保：
   - 结构的完整性
   - 节奏的合理性
   - 转折的自然性
   - 伏笔的合理性

### 使用场景
- 角色和世界观设定完成后
- 情节创作前进行规划
- 创作过程中需要调整时

### 输出数据结构
```yaml
outline:
  framework: {整体框架}
  key_turning_points: {关键转折点}
  chapter_plan: {章节规划}
  subplots: {支线规划}
  foreshadowings: {伏笔线索}
  conflict_progression: {冲突升级路径}
```

### 与其他节点的关系
- **依赖于**: 世界观设定、角色档案
- **输出给**: 场景描写、章节写作、编辑修订

## 4. 场景描写节点 (scene.txt)

### 设计目标
创作生动、有画面感的场景，让读者身临其境，为故事注入情感和氛围。

### 核心设计思路

**为什么这样设计？**

场景是故事的血肉，它能：
- 营造沉浸感和代入感
- 传达情绪和氛围
- 推动情节发展
- 展现角色性格

**设计决策：**

1. **五感描写法**
   - 视觉：色彩、光影、空间
   - 听觉：声音、节奏、音调
   - 嗅觉：气味、气息
   - 触觉：温度、质感
   - 时间与天气：环境条件

   五感并用，营造全方位的感官体验。

2. **氛围营造层次**
   - 情绪基调的设定
   - 张力或放松感的营造
   - 预兆与暗示
   - 情感的递进变化

3. **不同场景类型的特殊要求**
   - 战斗场景：动作节奏、危险感
   - 对话场景：角色性格、信息传递
   - 探索场景：新奇感、危险信号
   - 日常场景：生活气息、情感流露

4. **参数化设计**
   - `{worldview_context}`：环境基础
   - `{character_context}`：角色行动
   - `{scene_type}`：场景类型
   - `{setting}`：地点设定
   - `{mood}`：氛围要求
   - `{purpose}`：场景目的
   - `{scene_length}`：篇幅控制

5. **质量检查清单**
   包含8项自检指标，确保：
   - 五感的丰富性
   - 氛围的准确性
   - 细节的具体性
   - 节奏的合理性

### 使用场景
- 章节创作时，需要场景素材
- 氛围需要强化时
- 角色需要展现时

### 输出数据结构
```yaml
scene:
  scene_info: {场景信息}
  environment: {环境设定}
  atmosphere: {氛围描写}
  character_actions: {角色行动}
  full_text: {完整场景文本}
```

### 与其他节点的关系
- **依赖于**: 世界观设定、角色档案、大纲结构
- **输出给**: 章节写作、编辑修订

---

## 5. 章节写作节点 (chapter.txt)

### 设计目标
将所有设定和规划转化为完整的章节正文，产出可读的故事内容。

### 核心设计思路

**为什么这样设计？**

章节是故事的成品，它需要：
- 整合所有前置设定
- 推进情节发展
- 保持文风统一
- 满足读者期待

**设计决策：**

1. **章节四段式结构**
   - 开篇设计：衔接与引入
   - 情节推进：发展与冲突
   - 角色塑造：性格展现
   - 结尾设计：收束与钩子

   标准化的结构便于控制和检查。

2. **六个写作要素**
   - 情节推进：符合大纲
   - 角色塑造：展现性格
   - 细节描写：丰富场景
   - 节奏控制：张弛有度
   - 对话设计：符合角色
   - 心理描写：深化角色

3. **写作风格指南**
   - 冷峻风格：简洁有力
   - 幽默风格：风趣生动
   - 诗意风格：优美细腻
   - 简洁风格：明快直接

4. **参数化设计**
   - `{worldview_context}`：设定基础
   - `{character_context}`：角色行动
   - `{outline_context}`：情节规划
   - `{previous_context}`：上一章结尾
   - `{chapter_num}`：章节编号
   - `{chapter_title}`：章节标题
   - `{chapter_purpose}`：章节目的
   - `{target_length}`：目标字数
   - `{writing_style}`：写作风格
   - `{pov}`：叙事视角

5. **质量检查清单**
   包含9项自检指标，确保：
   - 大纲完成度
   - 衔接自然性
   - 角色一致性
   - 设定一致性
   - 风格统一性
   - 节奏流畅性
   - 细节生动性
   - 结尾吸引力
   - 语言正确性

### 使用场景
- 大纲规划完成后，正式创作
- 按章节顺序或非线性创作
- 章节需要重写时

### 输出数据结构
```yaml
chapter:
  chapter_info: {章节信息}
  outline_review: {大纲回顾}
  previous_ending: {上一章结尾}
  full_text: {章节正文}
  statistics:
    word_count: {实际字数}
    core_plot: {核心情节}
    character_growth: {角色变化}
    foreshadowings: {伏笔线索}
    transition_points: {衔接要点}
```

### 与其他节点的关系
- **依赖于**: 世界观设定、角色档案、大纲结构、场景描写
- **输出给**: 编辑修订、下一章的章节写作

---

## 6. 编辑修订节点 (revise.txt)

### 设计目标
从多维度优化文本质量，确保最终产出的专业性和可读性。

### 核心设计思路

**为什么这样设计？**

修订是故事的打磨过程，它能：
- 提升文本质量
- 消除错误和矛盾
- 优化阅读体验
- 确保出版标准

**设计决策：**

1. **五维检查系统**
   - 语法与用词：基础准确性
   - 一致性检查：前后统一
   - 文风统一性：风格连贯
   - 流畅性与节奏：阅读体验
   - 逻辑合理性：情节严密

   全面覆盖文本质量的所有维度。

2. **修订分级制度**
   - 轻度修改：仅修正明显错误
   - 中度修改：优化表达和结构
   - 深度修改：重构文本

   灵活适应不同修订需求。

3. **问题分类管理**
   - 必须修改：错误类型
   - 建议修改：优化类型
   - 可选优化：提升类型

   帮助作者把握修改优先级。

4. **参数化设计**
   - `{text_to_revise}`：待修订文本
   - `{worldview_context}`：设定核对
   - `{character_context}`：角色核对
   - `{check_types}`：检查维度
   - `{target_style}`：目标风格
   - `{revision_level}`：修订级别

5. **质量检查清单**
   包含8项自检指标，确保：
   - 检查维度完整性
   - 修改的适度性
   - 定位的准确性
   - 理由的充分性
   - 分类的合理性

### 使用场景
- 章节创作完成后进行修订
- 多轮迭代优化
- 不同版本的对比修订

### 输出数据结构
```yaml
revision:
  revision_info: {修订信息}
  issues:
    grammar: {语法与用词问题}
    consistency: {一致性问题}
    style: {文风问题}
    flow: {流畅性问题}
    logic: {逻辑问题}
  statistics:
    total_issues: {总问题数}
    must_fix: {必须修改数}
    should_fix: {建议修改数}
    optional: {可选优化数}
  revised_text: {修订后文本}
  revision_notes: {修订说明}
```

### 与其他节点的关系
- **依赖于**: 世界观设定、角色档案、章节写作
- **输出给**: 最终成品、下一轮修订

---

## 节点间的数据流

```
worldview (世界观设定)
    ↓
character (角色档案)
    ↓
outline (大纲结构)
    ↓
scene (场景描写) ─────┐
    ↓                  │
chapter (章节写作) ←───┘
    ↓
revise (编辑修订)
    ↓
final output (最终成品)
```

**说明：**
1. 场景描写节点是可选的，可独立调用
2. 章节写作可以多次迭代
3. 编辑修订可以多轮进行
4. 每个节点都可以独立调用进行局部修改

---

## 参数化机制

所有提示词都支持参数替换，使用 `{parameter_name}` 格式。参数分为两类：

### 1. 上下文参数
从上游节点继承的参数：
- `{worldview_context}`：世界观设定
- `{character_context}`：角色设定
- `{outline_context}`：大纲规划
- `{previous_context}`：前文内容

### 2. 配置参数
用户自定义的参数：
- `{genre}`, `{theme}`, `{tone}`：故事配置
- `{total_chapters}`, `{pacing}`：结构配置
- `{target_length}`, `{writing_style}`：写作配置
- `{revision_level}`, `{check_types}`：修订配置

### 参数替换规则
1. 参数用 `{}` 包裹
2. 参数名使用下划线分隔单词
3. 未提供的参数使用默认值
4. 参数值为空时，相关部分可省略

---

## 质量保障体系

每个提示词都内置质量检查清单，包含：

### 1. 强制检查项
- 必须满足的最低质量标准
- 发现问题必须修改

### 2. 建议检查项
- 提升质量的优化建议
- 根据具体情况决定是否采纳

### 3. 评分机制
- 通过/不通过的判断标准
- 多维度质量评分

---

## 扩展性设计

提示词模板预留了扩展空间：

1. **新增检查维度**
   - 在质量检查清单中添加新项目
   - 在输出格式中添加新字段

2. **新参数支持**
   - 在参数列表中添加新参数
   - 在提示词正文中引用新参数

3. **输出格式调整**
   - 修改 YAML 结构
   - 添加新的输出字段

---

## 使用建议

1. **顺序使用**
   按照世界观 → 角色 → 大纲 → 场景 → 章节 → 修订的顺序，确保信息完整传递。

2. **迭代优化**
   任何节点都可以重复调用，进行多轮迭代优化。

3. **模块化复用**
   场景描写可以单独调用，用于特定场景的素材准备。

4. **参数定制**
   根据具体项目需求，调整参数值以适应不同类型的创作。

5. **版本管理**
   每次修订后保存版本，便于对比和回滚。

---

## 总结

StoryFlow 提示词系统的核心优势：

1. **结构化**: 清晰的输入输出格式，便于自动化处理
2. **参数化**: 灵活的参数配置，适应不同创作需求
3. **模块化**: 独立的节点设计，支持灵活组合
4. **质量化**: 内置质量检查，保障输出质量
5. **可扩展**: 预留扩展空间，支持功能增强

这套提示词系统为 AI 辅助小说创作提供了完整的流程框架和质量保障。
