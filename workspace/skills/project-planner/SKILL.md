# project-planner

**【精算天命】Act on Fate** — 

## 目的

将模糊的想法转化为完整的、可执行的项目规格文档。

**输出是规格文档，不是代码，不是实现计划。** 只捕获 WHAT 和 WHY，不涉及 HOW。

## 内网兼容性

- ✅ `read` - 读取项目文件、文档
- ✅ `write` - 写入规格文档
- ✅ `edit` - 修改文档
- ✅ `exec` - 读取git历史等（可选）
- ✅ 所有功能完全可用

## 工作流程

```
探索上下文 → 评估范围 → 澄清问题 → 提出方向 → 起草规格 → 自我审查 → 用户审核 → 写入磁盘
```

### 步骤说明

1. **探索上下文** — 读取项目文件、文档、git历史、现有规格
2. **评估范围** — 这是一个规格还是需要分解？
3. **澄清问题** — 一次一个问题，追问到底
4. **提出方向** — 提出2-3个高层产品方向及权衡
5. **起草规格** — 写结构化的规格文档
6. **自我审查** — 检查完整性、一致性、是否有实现泄漏
7. **用户审核** — 展示给用户批准，如有需要迭代
8. **写入磁盘** — 保存到 `docs/specs/YYYY-MM-DD-<topic>.md`

**最终状态是书面规格。** 这个技能不会过渡到实现、规划或其他技能。用户决定如何使用规格。

## 提问技巧

### 如何提问

- **从开放开始** — 让用户倾倒他们的心理模型，不要用结构打断
- **跟随能量** — 用户强调什么，就深入什么
- **挑战模糊** — 永不接受模糊答案。"好"意味着什么？"用户"是谁？"简单"意味着多简单？
- **把抽象变具体** — "给我演示一下怎么用""举个例"
- **澄清歧义** — "你说Z，是指A还是B？"
- **知道何时停止** — 当理解了做什么、为什么做、谁来做、完成是什么样的时候，提出继续

### 提问机制

- 每次一条消息一个问题。如果一个主题需要更多，分成多条消息
- 尽量使用多选题——对具体选项的反应比对开放式提示更容易
- 当用户选择"其他"或想自由解释时，切换到纯文本
- 2-4个选项是理想的。永远不要用通用类别

### 要问什么

| 类别 | 示例问题 |
|------|---------|
| 动机 | "是什么促使这个？""你现在用什么代替？" |
| 具体性 | "给我演示一下""举个例" |
| 澄清 | "你说X，是指A还是B？" |
| 成功 | "你怎么知道它在运行？""完成是什么样？" |
| 边界 | "这个明确不是什么？" |

### 禁止问什么

- 技术实现细节（那是规划的事）
- 架构模式（那是规划的事）
- 用户的专业技术水平（无关——系统来构建）
- 成功指标（从工作中推断）
- 与上下文无关的预设问题

### 背景检查清单

（在脑中检查，不是大声说出来）：

- [ ] 用户在构建什么（足够具体，能向陌生人解释）
- [ ] 为什么需要存在（驱动它的难题或欲望）
- [ ] 谁来用（即使只是自己）
- [ ] "完成"什么样（可观察的结果）

当四件事都清楚时，提出继续。如果用户想继续探索，就继续。

## 规格格式

规格有这些部分：

1. **Core Value（核心价值）** — 一句话，最重要的事
2. **Problem Statement（问题陈述）** — 什么问题，谁有，为什么现在
3. **Requirements（需求）** — 必须有、应该有、超出范围（带原因）
4. **Constraints（约束）** — 带理由的硬限制
5. **Key Decisions（关键决策）** — 头脑风暴期间做出的决策及考虑的替代方案
6. **Reference Points（参考点）** — "我想要像X一样"的时刻、外部文档、灵感
7. **Open Questions（开放问题）** — 需要未来研究的未解决项目
8. **Future Considerations（未来考虑）** — 出现的但属于后期阶段的想法

### 需求必须具体可测试

| 好的需求 | 坏的需求 |
|---------|---------|
| "用户可以撤销最近10个操作" | "好的撤销支持" |
| "页面在3G下2秒内加载" | "快速性能" |
| "配合屏幕阅读器工作" | "可访问" |
| "导出为CSV和JSON" | "多种导出格式" |

## 关键原则

1. 每次一个问题 — 不要让人应接不暇
2. 跟随线程 — 不要走检查清单
3. YAGNI严格 — 删除任何不明显需要的东西
4. 具体决策 — "卡片布局"不是"现代简洁"
5. 不涉及实现 — WHAT和WHY，从不HOW
6. 捕获一切 — 范围外的想法进入Future Considerations，从不丢失
7. 增量验证 — 在继续之前确认理解
8. 规格独立 — 任何人都应该能阅读它并理解项目

## 内网使用提示

1. **项目文件路径** - 如有现有项目，提供目录路径
2. **参考文档** - 任何相关的现有文档/规格
3. **保存位置** - 规格默认保存到 `docs/specs/`，可自定义路径
