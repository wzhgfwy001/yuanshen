## 2026-04-25 StoryFlow项目失败

### 问题现象
StoryFlow项目始终无法正常运行，Agent 1执行失败返回"No output"

### 因果链分析

```
问题：Agent 1返回"No output"
    ↓
直接原因：WorldBuildingNode.execute()未获得有效输出
    ↓
背后原因1：MiniMax API调用失败（API Key配置问题 - nodes.py查找STORYFLOW_API_KEY，但实际是MINIMAX_API_KEY）
背后原因2：调试日志无法查看（print语句未出现在后端日志）
    ↓
背后原因1的根本原因：代码写死后使用 STORYFLOW_API_KEY 环境变量，未做兼容处理
背后原因2的根本原因：Flask-SocketIO + gevent 模式下 stdout 缓冲/重定向问题
    ↓
更根本原因：Windows环境下gevent与Flask-SocketIO兼容性未验证就部署
    ↓
最根本原因：急于求成，没有先在小范围验证就开始大规模开发
```

### 预防措施

| 层级 | 预防措施 |
|------|---------|
| 直接原因 | API调用添加详细的try-catch和日志记录 |
| 背后原因1 | 添加环境变量兼容性：优先读STORYFLOW_API_KEY，fallback到MINIMAX_API_KEY |
| 背后原因2 | 调试日志写文件而不是依赖stdout |
| 更根本原因 | 复杂技术栈（WebSocket + gevent）在Windows环境先做技术验证 |
| 最根本原因 | 采用"先验证再扩展"的开发模式 |

### 决策点记录
- **错误决策：** 在未验证WebSocket可行性的情况下开始完整开发
- **正确做法：** 先创建一个最简单的WebSocket测试，验证能工作后再增加功能

## 2026-04-24 Playwright MCP失败

### 问题现象
Microsoft Playwright MCP无法驱动Edge浏览器，只能驱动Chrome

### 因果链分析

```
问题：用户要求用Playwright MCP打开Edge，但实际只支持Chrome
    ↓
直接原因：Playwright MCP硬编码Chrome路径，不支持Edge
    ↓
背后原因：创建了复杂的bridge架构试图绕过限制
    ↓
根本原因：没有先阅读Playwright MCP的文档或源码就动手
    ↓
最根本原因：不愿意说"我不知道怎么实现"，而是试图自己想办法绕过
```

### 预防措施

| 层级 | 预防措施 |
|------|---------|
| 直接原因 | 先确认工具的能力范围，不要假设 |
| 背后原因 | 遇到限制时先停下来问用户"这个限制怎么处理"，而不是自己绕路 |
| 根本原因 | 新工具接入前先读SKILL.md和相关文档 |
| 最根本原因 | 不懂就问，不要假装懂 |

### 决策点记录
- **错误决策：** 创建复杂的bridge架构试图让Playwright MCP使用Edge
- **正确做法：** 确认工具局限性后，直接告知用户并提供替代方案
