# 瓶颈诊断器 (Bottleneck Diagnosis)

## 核心定位

借鉴 Startup Coach Workshop 的核心理念，在任务分类之前增加问题诊断层，识别用户问题的真正瓶颈所在。

## 设计原则

**单一入口原则：** "一次解决一个主要瓶颈"

```
用户问题 → 瓶颈诊断 → 任务分类 → 解决方案
              ↓
        当前主要瓶颈
        优先级排序
        解决建议
```

## 问题层次模型

### L1: 需求不清 (Unclear Requirements)

**定义：** 用户自己不清楚真正想要什么

**特征：**
- 描述模糊、抽象
- "做个东西"、"弄一下"
- 反问"你觉得呢"
- 多个互相矛盾的表述

**诊断问题：**
```
1. 你想解决什么问题？
2. 这个问题的最终目标是什么？
3. 谁会使用这个结果？
4. 如何判断成功？
```

**处理策略：** 澄清式提问，不直接执行

---

### L2: 范围模糊 (Unclear Scope)

**定义：** 知道要做什么，但边界不清

**特征：**
- "等等，还有..."
- 不断添加新需求
- 范围无明确边界
- 无法估算工作量

**诊断问题：**
```
1. 这个任务的核心是什么？
2. 包含哪些部分？
3. 不包含哪些部分？
4. 有没有大小/长度限制？
```

**处理策略：** 帮助明确范围，获取清晰边界

---

### L3: 实施模糊 (Unclear Implementation)

**定义：** 知道做什么，但不知道怎么做

**特征：**
- "帮我写个程序"
- "做个设计方案"
- 技术细节不明确
- 实现路径不清晰

**诊断问题：**
```
1. 有没有参考资料/示例？
2. 技术栈有偏好吗？
3. 有没有现成方案可以参考？
4. 预期采用什么技术路线？
```

**处理策略：** 提供实施路径建议，或请求更多信息

---

### L4: 产品空泛 (Vague Product)

**定义：** 有方向，但产品形态不具体

**特征：**
- "做个App"
- "做个网站"
- 功能模块不明确
- 用户场景不清晰

**诊断问题：**
```
1. 目标用户是谁？
2. 核心功能有哪些？
3. 与现有产品的差异？
4. 期望的交付形式？
```

**处理策略：** 产品设计引导，或加载产品设计模板

---

### L5: UX不全 (Incomplete UX)

**定义：** 有产品，但体验不完整

**特征：**
- "做个登录功能"
- 交互细节缺失
- 异常情况未考虑
- 用户流程不完整

**诊断问题：**
```
1. 用户的操作流程是什么？
2. 异常/错误情况如何处理？
3. 需要哪些反馈提示？
4. 有无权限/状态管理需求？
```

**处理策略：** UX补全，或加载UX审查模板

---

### L0: 执行明确 (Ready to Execute)

**定义：** 问题清晰，可以直接执行

**特征：**
- 任务描述完整
- 边界清晰
- 有具体要求
- 无歧义

**处理策略：** 直接进入任务分类

## 诊断流程

### 步骤1: 问题层次识别

```
输入：用户任务描述
输出：检测到的问题层次

检测方法：
1. 关键词匹配（"做个"、"帮我"、"不是很清楚"）
2. 约束条件分析（有无具体要求）
3. 完整性检查（5W1H覆盖度）
```

### 步骤2: 瓶颈优先级

```
当前层级 = 最高未解决问题的层级

优先级规则：
1. L1 > L2 > L3 > L4 > L5 > L0
2. 低层级问题未解决，不处理高层级
3. 一次只关注一个主要瓶颈
```

### 步骤3: 诊断输出

```json
{
  "diagnosis": {
    "current-level": "L2",
    "level-name": "范围模糊",
    "description": "任务边界不清晰，需要明确范围",
    "confidence": 0.85,
    "detected-signals": [
      "描述中包含'等等还有'",
      "缺少具体范围定义"
    ],
    "clarifying-questions": [
      "这个任务的核心交付物是什么？",
      "包含哪些具体内容？"
    ],
    "recommendation": "MUST_CLARIFY"
  },
  "task-readiness": "NOT_READY",
  "blocking-issue": "scope-ambiguity"
}
```

## 任务就绪状态

| 状态 | 含义 | 处理 |
|------|------|------|
| READY | 问题清晰 | 进入任务分类器 |
| NEEDS_CLARIFICATION | 需要澄清 | 输出澄清问题 |
| PARTIALLY_READY | 部分清晰 | 处理清晰部分，标注模糊部分 |

## 诊断器内部逻辑

### 主要函数

```javascript
// 诊断主函数
async function diagnose(taskDescription) {
  // 1. 问题层次识别
  const level = identifyLevel(taskDescription);
  
  // 2. 瓶颈优先级
  const bottleneck = prioritizeBottleneck(level);
  
  // 3. 生成澄清问题
  const questions = generateClarifyingQuestions(bottleneck);
  
  // 4. 评估任务就绪状态
  const readiness = evaluateReadiness(bottleneck);
  
  return {
    currentLevel: level,
    bottleneck,
    questions,
    readiness,
    recommendation: getRecommendation(readiness)
  };
}
```

## 置信度评估

| 层级 | 基础置信度 | 调整因素 |
|------|-----------|---------|
| L1 | 0.80 | +0.1 有反问，-0.1 有具体目标 |
| L2 | 0.75 | +0.1 有关键词，-0.1 边界清晰 |
| L3 | 0.70 | +0.1 无参考，-0.1 有示例 |
| L4 | 0.70 | +0.1 无用户，-0.1 用户明确 |
| L5 | 0.65 | +0.1 缺流程，-0.1 流程完整 |
| L0 | 0.90 | 无需调整 |

---

## 异常处理

### 无法诊断

**现象：** 所有层级置信度都<0.5

**处理：** 返回 L1（需求不清），要求澄清

### 多层级同时存在

**原则：** 只报告最高优先级（数字最小）的层级

**示例：** 同时检测到L2和L4 → 只报告L2
